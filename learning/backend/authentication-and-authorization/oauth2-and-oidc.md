# OAuth2 & OpenID Connect

Delegated authorization flows enabling secure third-party integrations, microservice access, and single sign-on (SSO) systems.

---

## Why It Matters

Exposing user credentials directly to client applications is a critical anti-pattern. OAuth 2.0 resolves this by introducing token-based delegated authorization, allowing users to grant limited API access to third-party applications without disclosing passwords. OpenID Connect (OIDC) extends this by adding an identity layer, resolving a historic limitation of OAuth 2.0: using authorization tokens to prove user identity. Using insecure flows—such as the deprecated Implicit Flow or failing to implement Proof Key for Code Exchange (PKCE) in public clients—leaves applications vulnerable to credential interception and authorization code hijacking.

---

## Core Concepts

OAuth 2.0 defines four primary system roles:

1. **Resource Owner**: The end-user who grants access to a protected resource.
2. **Client**: The application making requests on behalf of the Resource Owner.
3. **Authorization Server**: The server that authenticates the user and issues tokens (e.g., Okta, Auth0, Keycloak).
4. **Resource Server**: The API hosting the user's data, which validates access tokens.

### 1. Authorization Code Flow with PKCE

For public clients like Single Page Applications (SPAs) or Mobile Apps, client secrets cannot be securely stored. The **Authorization Code Flow with PKCE** replaces static client secrets with dynamic cryptographic verification.

```
+-------------+                     +----------------------+
|             |--(A)- Authorization ---->|                      |
|             |       Request with PKCE      |                      |
|             |       Challenge              |                      |
|             |<-(B)- Auth Code ---------|                      |
|   Client    |                          |                      |
|  (SPA/App)  |--(C)- Auth Code + ------->| Authorization Server |
|             |       Verifier Check         |                      |
|             |                              |                      |
|             |<-(D)- Tokens (ID & Access)---|                      |
|             |                              |                      |
+-------------+                              +----------------------+
```

- **Code Verifier**: A high-entropy cryptographic random string generated by the client.
- **Code Challenge**: A SHA-256 hash of the verifier, Base64URL-encoded.
- **The Handshake**:
  1. The client redirects the user to the Authorization Server, passing the **Code Challenge** and hashing method.
  2. The user authenticates and the Authorization Server returns a short-lived **Authorization Code** to the client.
  3. The client calls the Token Endpoint directly, sending the **Authorization Code** along with the raw **Code Verifier**.
  4. The Authorization Server hashes the verifier and validates it against the stored challenge. If they match, it issues the tokens.

### 2. Client Credentials Flow

Used for machine-to-machine (M2M) communication (e.g., a backend cron service calling a billing API). Because there is no end-user, the client authenticates itself directly to the Authorization Server using a secure `client_id` and `client_secret` to obtain an access token.

### 3. OpenID Connect (OIDC) vs. OAuth 2.0

OAuth 2.0 is strictly for authorization (access delegating). OIDC layers user identity on top:

| Feature          | OAuth 2.0                                                                                   | OpenID Connect (OIDC)                                                                           |
| :--------------- | :------------------------------------------------------------------------------------------ | :---------------------------------------------------------------------------------------------- |
| **Primary Goal** | **Authorization**: Grant permission to access resources.                                    | **Authentication**: Verify who the user is.                                                     |
| **Token Type**   | **Access Token**: Typically an opaque string or JWT intended for the API (Resource Server). | **ID Token**: Always a signed JWT containing profile claims (e.g., name, email) for the Client. |
| **Discovery**    | Ad-hoc endpoint configuration per provider.                                                 | Standardized dynamic config at `.well-known/openid-configuration`.                              |

## Real-World Production Learnings

When architecting a frontend SPA integrated with a microservice backend, we initially implemented the legacy **Implicit Flow**, where the client received the Access Token directly in the URL redirect hash fragment.

During penetration testing, we discovered that:

1. Access tokens were leaking into web server access logs via the HTTP `Referer` header.
2. Web browser history stored the exact token strings, leaving them vulnerable to theft via physical access or browser-level scripting exploits.

We refactored our system to use the **Authorization Code Flow with PKCE** in conjunction with the **BFF (Backend-for-Frontend) Pattern**:

- The SPA never sees or stores Access or Refresh tokens.
- A secure Node.js middleware server (the BFF) initiates the PKCE authorization flow, handles the authorization code exchange, and stores the tokens in a secure server-side session.
- The BFF communicates with the SPA using secure, `HttpOnly`, `SameSite=Strict` encrypted session cookies.
- When the SPA requests data from microservices, the BFF interceptor decrypts the session cookie, extracts the backend Access Token, appends it as a Bearer token, and forwards it to the API gateway.

This architectural shift decoupled token storage from the client, securing our tokens from cross-site scripting (XSS) risks.
